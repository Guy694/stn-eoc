
ให้แก้ไขจากโครงสร้างหน้าเดิมเท่านั้น ห้ามรื้อ Layout หลักของระบบเดิม ให้เพิ่ม Disease Outbreak Dashboard เป็น Section ใหม่ในหน้าแรก โดยคง Sidebar, Map, Header และ Theme เดิมไว้ แล้วเพิ่มกราฟและข้อมูลโรคไข้เลือดออกในรูปแบบ Component แยก เช่น DiseaseSummaryCards, WeeklyTrendChart, DistrictCaseChart, DistrictWeeklyHeatmap, PatientTypeChart, AgeSexChart, DiseaseCodeTable และ PublicAdvicePanel เพื่อให้ดูแลรักษาโค้ดง่าย
ปรับปรุงหน้าหลักของระบบศูนย์ปฏิบัติการภาวะฉุกเฉินจังหวัดสตูล หรือ EOC Dashboard ให้รองรับกรณี “เปิด EOC โรคระบาด” โดยขณะนี้เป็นเหตุการณ์ “โรคไข้เลือดออก” ให้หน้าแรกแสดงข้อมูลโรคระบาดในรูปแบบ Dashboard เชิงสาธารณะ เน้นการสื่อสารสถานการณ์ ภาพรวมแนวโน้ม และการติดตามความรุนแรงของสถานการณ์ในระดับจังหวัดและระดับอำเภอ

ให้พัฒนา/ปรับ UI โดยยึดรูปแบบเดิมของระบบ EOC ที่มีแผนที่จังหวัดสตูล, Sidebar Menu, Summary Cards และ Dashboard กลาง แต่เพิ่มส่วนข้อมูลโรคระบาดเป็นโมดูลหลักในหน้าแรก

## เป้าหมายของหน้าหลัก

1. แสดงสถานะการเปิด EOC โรคระบาด
2. แสดงข้อมูลสถานการณ์โรคไข้เลือดออกแบบเข้าใจง่าย
3. แสดงแนวโน้มผู้ป่วยรายสัปดาห์ของแต่ละปี
4. เปรียบเทียบจำนวนผู้ป่วยระหว่างปี
5. แสดงจำนวนผู้ป่วยแยกรายอำเภอ
6. แสดงข้อมูลประเภทผู้ป่วย กลุ่มโรค รหัสโรค และตัวชี้วัดที่จำเป็น
7. ใช้ข้อมูลเฉพาะระดับสาธารณะ ห้ามแสดงข้อมูลส่วนบุคคล เช่น ชื่อ นามสกุล เลขบัตรประชาชน เบอร์โทร หรือพิกัดบ้านรายบุคคล

## โครงสร้างหน้าหลักที่ต้องการ

### 1. Header: สถานะการเปิด EOC

ให้แสดงแถบด้านบนของหน้า Dashboard ประกอบด้วย

* ชื่อเหตุการณ์: EOC โรคไข้เลือดออก จังหวัดสตูล
* ประเภทเหตุการณ์: โรคระบาด / Communicable Disease
* วันที่เปิด EOC
* ระยะเวลาที่เปิด EOC แล้ว เช่น เปิดมาแล้ว 7 วัน 4 ชั่วโมง
* ระดับสถานการณ์ เช่น เฝ้าระวัง / ระบาด / วิกฤต
* วันที่อัปเดตข้อมูลล่าสุด
* ปุ่มเลือกปีงบประมาณหรือปี พ.ศ.
* ปุ่มเลือกอำเภอ
* ปุ่มเลือกช่วงสัปดาห์
* ปุ่ม Export เฉพาะข้อมูลสาธารณะ เช่น PNG, PDF, CSV

### 2. Summary Cards ภาพรวมจังหวัด

ให้สร้างการ์ดสรุปข้อมูลสำคัญด้านบนของ Dashboard อย่างน้อย 8 การ์ด ได้แก่

1. ผู้ป่วยสะสมปีปัจจุบัน
2. ผู้ป่วยรายใหม่สัปดาห์ล่าสุด
3. จำนวนผู้ป่วยเพิ่ม/ลดจากสัปดาห์ก่อน
4. อัตราป่วยต่อแสนประชากร
5. จำนวนผู้เสียชีวิต
6. อัตราป่วยตาย
7. อำเภอที่พบผู้ป่วยสูงสุด
8. จำนวนอำเภอที่มีรายงานผู้ป่วย

รูปแบบการ์ดควรใช้สีตามระดับความรุนแรง เช่น เขียว = ปกติ, เหลือง = เฝ้าระวัง, ส้ม = ระบาด, แดง = วิกฤต

### 3. กราฟแนวโน้มรายสัปดาห์

เพิ่มกราฟหลักเป็น Line Chart หรือ Area Chart แสดงจำนวนผู้ป่วยโรคไข้เลือดออก “รายสัปดาห์” โดยต้องสามารถดูข้อมูลได้หลายปีในกราฟเดียว

ข้อกำหนดกราฟ:

* แกน X = สัปดาห์ที่ 1–52 หรือ 1–53
* แกน Y = จำนวนผู้ป่วย
* เส้นกราฟแยกตามปี เช่น ปี 2566, 2567, 2568, 2569
* แสดงเส้นปีปัจจุบันให้เด่นที่สุด
* มี Tooltip แสดง ปี, สัปดาห์, จำนวนผู้ป่วย, จำนวนเพิ่ม/ลดจากสัปดาห์ก่อน
* มี Legend ให้เปิด/ปิดแต่ละปีได้
* มีเส้นค่าเฉลี่ยย้อนหลัง 3 ปี หรือ Median 5 ปี ถ้ามีข้อมูล
* แสดงจุดเตือนเมื่อจำนวนผู้ป่วยเกินค่ามาตรฐานหรือเกินค่าเฉลี่ยย้อนหลัง

หัวข้อกราฟ:
“แนวโน้มผู้ป่วยโรคไข้เลือดออก รายสัปดาห์ เปรียบเทียบรายปี”

### 4. กราฟจำนวนผู้ป่วยรายอำเภอ

สร้าง Bar Chart หรือ Horizontal Bar Chart สำหรับแสดงจำนวนผู้ป่วยสะสมแยกรายอำเภอ

ข้อมูลที่ต้องแสดง:

* เมืองสตูล
* ควนโดน
* ควนกาหลง
* ท่าแพ
* ละงู
* ทุ่งหว้า
* มะนัง

ข้อกำหนดกราฟ:

* เรียงจากอำเภอที่มีจำนวนผู้ป่วยมากไปน้อย
* แสดงจำนวนผู้ป่วยและอัตราป่วยต่อแสนประชากร
* ใช้สีตามระดับความรุนแรงของแต่ละอำเภอ
* เมื่อกดที่อำเภอ ให้ Dashboard ส่วนอื่นกรองข้อมูลตามอำเภอนั้น
* มีปุ่มดูแบบ “จำนวนผู้ป่วย” และ “อัตราป่วยต่อแสนประชากร”

หัวข้อกราฟ:
“จำนวนผู้ป่วยโรคไข้เลือดออกสะสม แยกรายอำเภอ”

### 5. กราฟ Heatmap รายอำเภอ x รายสัปดาห์

เพิ่ม Heatmap เพื่อแสดงความเข้มของการระบาดในแต่ละอำเภอตามสัปดาห์

ข้อกำหนด:

* แกน X = สัปดาห์
* แกน Y = อำเภอ
* สีเข้มขึ้นเมื่อจำนวนผู้ป่วยมากขึ้น
* Tooltip แสดง อำเภอ, สัปดาห์, จำนวนผู้ป่วย, อัตราป่วย
* ใช้เพื่อให้ทีม EOC เห็น Pattern การระบาดรายพื้นที่อย่างรวดเร็ว

หัวข้อกราฟ:
“Heatmap การระบาดรายสัปดาห์ แยกรายอำเภอ”

### 6. กราฟประเภทผู้ป่วย

เพิ่มกราฟ Donut Chart หรือ Stacked Bar Chart สำหรับแสดงประเภทผู้ป่วย เช่น

* ผู้ป่วยสงสัย
* ผู้ป่วยเข้าข่าย
* ผู้ป่วยยืนยัน
* ผู้เสียชีวิต

ให้แสดงทั้งจำนวนและร้อยละ โดยเน้นว่านี่คือข้อมูลภาพรวม ไม่ใช่ข้อมูลรายบุคคล

หัวข้อกราฟ:
“ประเภทผู้ป่วยตามนิยามการเฝ้าระวัง”

### 7. กราฟกลุ่มอายุและเพศ

เพิ่มกราฟแสดงการกระจายผู้ป่วยตามกลุ่มอายุและเพศ โดยเป็นข้อมูลรวมเท่านั้น

กลุ่มอายุที่แนะนำ:

* 0–4 ปี
* 5–9 ปี
* 10–14 ปี
* 15–24 ปี
* 25–34 ปี
* 35–44 ปี
* 45–54 ปี
* 55–64 ปี
* 65 ปีขึ้นไป

รูปแบบกราฟที่แนะนำ:

* Bar Chart แยกกลุ่มอายุ
* Stacked Bar แยกเพศ
* Population Pyramid ถ้าเหมาะสม

หัวข้อกราฟ:
“การกระจายผู้ป่วยตามกลุ่มอายุและเพศ”

### 8. ข้อมูลรหัสโรคและกลุ่มโรค

เพิ่มตารางหรือ Card แสดงข้อมูลรหัสกลุ่มโรคที่เกี่ยวข้องกับไข้เลือดออก เช่น

* A90: Dengue fever
* A91: Dengue haemorrhagic fever
* กลุ่มโรค: โรคติดต่อนำโดยยุงลาย
* ประเภทโรค: โรคติดต่อที่ต้องเฝ้าระวัง
* หน่วยรายงาน: สถานพยาบาล / ระบบเฝ้าระวังโรค / งานควบคุมโรค

ตารางควรแสดง:

* รหัสโรค
* ชื่อโรคภาษาไทย
* ชื่อโรคภาษาอังกฤษ
* จำนวนผู้ป่วย
* ร้อยละของผู้ป่วยทั้งหมด
* แนวโน้มเทียบสัปดาห์ก่อน

หัวข้อ:
“รหัสโรคและกลุ่มโรคที่ใช้ในการเฝ้าระวัง”

### 9. แผนที่สถานการณ์โรคระบาด

## Logic ที่ต้องใช้ใน Dashboard

ให้คำนวณข้อมูลสำคัญดังนี้

1. จำนวนผู้ป่วยสะสม = ผลรวม total_cases ตามช่วงเวลาที่เลือก
2. ผู้ป่วยรายสัปดาห์ = ผลรวม new_cases ตาม epi_week
3. อัตราป่วยต่อแสนประชากร = จำนวนผู้ป่วย / ประชากร × 100,000
4. อัตราป่วยตาย = จำนวนผู้เสียชีวิต / จำนวนผู้ป่วย × 100
5. แนวโน้มเพิ่ม/ลด = เปรียบเทียบผู้ป่วยสัปดาห์ล่าสุดกับสัปดาห์ก่อน
6. ค่าเฉลี่ยย้อนหลัง = ค่าเฉลี่ยจำนวนผู้ป่วยรายสัปดาห์ย้อนหลัง 3 ปี หรือ 5 ปี
7. ระดับความเสี่ยง = กำหนดจากจำนวนผู้ป่วย อัตราป่วย และการเพิ่มขึ้นของแนวโน้ม

## Visual Design

ใช้ดีไซน์แบบ Government Emergency Dashboard ที่ดูเป็นทางการ ทันสมัย และอ่านง่าย

แนวทางสี:

* Navy / Deep Blue สำหรับโครงหลัก
* Cyan / Sky Blue สำหรับข้อมูลทั่วไป
* Yellow สำหรับเฝ้าระวัง
* Orange สำหรับระบาด
* Red สำหรับวิกฤต
* Green สำหรับสถานการณ์ปกติ
* White / Light Gray สำหรับพื้นหลังข้อมูล

สไตล์ UI:

* Card-based layout
* Dashboard grid
* Rounded corners
* Clear typography
* Icon ประกอบแต่ละหมวด
* ใช้ Chart ที่อ่านง่าย ไม่ตกแต่งเกินจำเป็น
* รองรับจอห้องบัญชาการ 16:9
* รองรับ Responsive สำหรับ Tablet และ Mobile

## Tech Requirement

ให้ใช้ Next.js / React และใช้ Recharts สำหรับกราฟหลัก เช่น

* LineChart สำหรับแนวโน้มรายสัปดาห์
* BarChart สำหรับรายอำเภอ
* AreaChart สำหรับเปรียบเทียบหลายปี
* PieChart หรือ Donut Chart สำหรับประเภทผู้ป่วย
* Heatmap อาจใช้ CSS Grid, Recharts Custom Cell หรือ Library เสริมที่เหมาะสม

หากยังไม่มี API จริง ให้สร้าง Mock Data ที่มีโครงสร้างเหมือนข้อมูลจริง และแยกไฟล์ mock data ออกมาให้ชัดเจน เพื่อให้สามารถเชื่อมต่อ API หรือฐานข้อมูลจริงในภายหลังได้
