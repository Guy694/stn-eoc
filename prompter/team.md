# Prompt: พัฒนาส่วนเมนูเจ้าหน้าที่ตามกลุ่มภารกิจ EOC

คุณคือ Senior Full-stack Developer, UX/UI Designer, GIS Dashboard Architect, Emergency Operation Center System Designer และผู้เชี่ยวชาญระบบบัญชาการเหตุการณ์ภัยพิบัติ

ให้พัฒนาส่วน "เมนูเจ้าหน้าที่" โดยยึดโครงสร้างการทำงานของ EOC จริง และรองรับหลายประเภทเหตุการณ์ (เช่น flood, disease, festival-accidents)

## 1) เป้าหมายงาน
- เมื่อเจ้าหน้าที่เข้าสู่ระบบและเลือก EOC Module แล้ว หน้าแรกต้องแสดงเมนูกลุ่มภารกิจที่ "เปิดใช้งานใน session นั้น" เท่านั้น
- แต่ละกลุ่มภารกิจต้องมีอย่างน้อย 2 เมนูย่อย
- Dashboard ของกลุ่มภารกิจ (สรุปสถานการณ์ด้วย KPI/Chart/Table)
- รายงานผลของกลุ่มภารกิจ (สร้าง/แก้ไข/ค้นหา/ส่งออก)

## 2) User Flow ที่ต้องได้
1. เจ้าหน้าที่ Login สำเร็จ
2. เลือกโมดูล EOC (เช่น อุทกภัย)
3. เข้าหน้า Overview ของ EOC Session ปัจจุบัน
4. ระบบโหลดรายการกลุ่มภารกิจที่ active ใน session นั้น
5. แสดง Team Menu Grid/Card ตามสิทธิ์ผู้ใช้
6. เจ้าหน้าที่เข้ากลุ่มภารกิจที่ตนรับผิดชอบ
7. ภายในกลุ่มภารกิจมี
- Dashboard
- รายงานผล
- (ถ้าจำเป็น) รายการงาน/สถานะการส่งข้อมูล/ประวัติการแก้ไข

## 3) ขอบเขตเชิงระบบที่ต้องออกแบบ
- Dynamic Menu จากฐานข้อมูล eoc_session_teams + eoc_teams + eoc_team_members
- RBAC ตาม role และ team membership
- รองรับสถานะ session (active/closed):
- active: เพิ่ม/แก้ไขรายงานได้
- closed: อ่านอย่างเดียว
- รองรับ Audit Log ทุกการสร้าง/แก้ไขรายงาน
- รองรับ Mobile-first และ Desktop

## 4) ร่างเมนูกลุ่มภารกิจทั้งหมด (สำหรับตรวจสอบความถูกต้อง)
ให้ใช้รายการนี้เป็น baseline ก่อนพัฒนา

1. COMMANDER - ผู้บัญชาการเหตุการณ์และรองผู้บัญชาการเหตุการณ์
2. SAT - กลุ่มภารกิจตระหนักรู้สถานการณ์
3. PLANNING - กลุ่มภารกิจการวางแผน
4. SCIENTIFIC - กลุ่มภารกิจวิชาการ
5. JIT - กลุ่มภารกิจปฏิบัติการสอบสวนควบคุมโรค
6. ACTIVE_SURV - กลุ่มภารกิจการเฝ้าระวังเชิงรุก
7. CASE_MGMT - กลุ่มภารกิจการจัดการและดูแลรักษาผู้ป่วย
8. LAB - กลุ่มภารกิจห้องปฏิบัติการด้านสาธารณสุข
9. RISKCOM - กลุ่มภารกิจสื่อสารความเสี่ยง
10. VACCINE - กลุ่มภารกิจบริหารจัดการและบริการวัคซีน
11. MCATT - กลุ่มภารกิจช่วยเหลือเยียวยาจิตใจผู้ประสบภาวะวิกฤต
12. POE - กลุ่มภารกิจด่านควบคุมโรคระหว่างประเทศ
13. QUARANTINE - กลุ่มภารกิจปฏิบัติการกักกันโรค
14. MERT - กลุ่มภารกิจปฏิบัติการฉุกเฉินทางการแพทย์
15. SeRHT - กลุ่มภารกิจปฏิบัติการด้านอนามัยสิ่งแวดล้อม
16. LOGISTICS - กลุ่มภารกิจสำรองวัสดุ เวชภัณฑ์ และส่งกำลังบำรุง
17. LEGAL - กลุ่มภารกิจกฎหมาย
18. FINANCE - กลุ่มภารกิจการเงินและงบประมาณ
19. STAFFING - กลุ่มภารกิจจัดสรรกำลังคนในภาวะฉุกเฉิน
20. ADMIN - กลุ่มภารกิจด้านบริหารและธุรการ
21. ITSUPPORT - กลุ่มภารกิจด้านเทคโนโลยีดิจิทัล
22. LIAISON - กลุ่มภารกิจประสานงานและเลขานุการ
23. EOC_MGMT - กลุ่มภารกิจการจัดการศูนย์ปฏิบัติการภาวะฉุกเฉิน

## 5) ตัวอย่างฟังก์ชันรายทีมที่ต้องมี (ขั้นต่ำ)
- LOGISTICS

- รายงานผล: สร้างใบเบิก, อนุมัติ, จ่ายออก, ประวัติการเบิก

- รายงานผล: เพิ่มข่าวประชาสัมพันธ์, แนบไฟล์/ลิงก์, บันทึกผลตอบรับ

## 6) ข้อกำหนด UX/UI
- หน้า Overview แสดง Team Cards แบบ responsive
- แสดง badge สถานะทีม: ส่งข้อมูลแล้ว/ค้างส่ง/ล่าช้า

- ทุกทีมใช้ layout มาตรฐานเดียวกัน: Header KPI + Tabs (Dashboard, รายงานผล, ประวัติ)
- แยกสีตาม team เพื่อให้จดจำง่าย แต่ต้องคุม accessibility contrast

- ห้ามใช้ mock data ใน production path
- เพิ่ม loading, empty state, error state ทุกหน้า

## 8) ผลลัพธ์ที่ต้องส่งกลับจากการพัฒนา

3. รายการ API ที่ต้องใช้ต่อทีม
4. โครงสร้างตาราง/ฟิลด์ที่ต้องเพิ่ม (ถ้ามี)
5. แผนการพัฒนาเป็นเฟส (MVP -> Full)

## 10) ร่างแบบฟอร์มรายงานเฉพาะ 23 กลุ่มภารกิจ (สำหรับตรวจสอบก่อนพัฒนาจริง)

### 10.1 หลักเกณฑ์ร่วมของทุกแบบฟอร์ม
- ทุกแบบฟอร์มผูกกับ `session_id`, `session_team_id`, `team_code` และผู้บันทึกจาก session login โดยระบบ ไม่รับค่าเหล่านี้จากผู้ใช้
- ส่วนหัวร่วม: วันที่และเวลารายงาน, รอบรายงาน/เวรปฏิบัติ, ระดับความเร่งด่วน, สถานะ (`draft`, `submitted`, `approved`, `returned`), ผู้รับผิดชอบ, สรุปสถานการณ์ และไฟล์แนบ/ลิงก์อ้างอิง
- ฟิลด์ที่เป็นสถานที่ต้องเลือกได้ทั้ง จังหวัด/อำเภอ/ตำบล/หมู่บ้าน และพิกัดแผนที่เมื่อเกี่ยวข้อง
- เมื่อ Session ปิด แบบฟอร์มและเอกสารแนบแก้ไขไม่ได้; ทุกการสร้าง แก้ไข ส่ง อนุมัติ หรือส่งกลับ ต้องบันทึก `activity_logs` พร้อม `session_id`, `team_code`, ชนิดเอกสาร และเลขอ้างอิง
- รายการรายงานทุกทีมต้องมี: ค้นหา, กรองช่วงวัน/สถานะ/ผู้รับผิดชอบ, เปิดดูรายละเอียด, ประวัติการแก้ไข และส่งออก CSV/XLSX หรือ PDF ตามสิทธิ์

### 10.2 COMMANDER - รายงานสั่งการเหตุการณ์
**วัตถุประสงค์:** บันทึกการตัดสินใจ คำสั่ง และการติดตามข้อสั่งการของผู้บัญชาการเหตุการณ์

**ส่วนข้อมูลหลัก:** เลขที่คำสั่ง, วันเวลาเริ่มมีผล, ระดับเหตุการณ์, สรุปสถานการณ์เพื่อการตัดสินใจ, คำสั่ง/มอบหมายงาน, ทีมผู้รับมอบ, ผู้รับผิดชอบ, กำหนดแล้วเสร็จ, ทรัพยากรที่อนุมัติ, ผู้อนุมัติ และสถานะติดตามผล

**รายการติดตาม:** ผลการดำเนินการ, วันที่รายงานกลับ, อุปสรรค, การตัดสินใจเพิ่มเติม, สถานะ `open/in_progress/completed/cancelled`

### 10.3 SAT - รายงานสถานการณ์ (Situation Report)
**วัตถุประสงค์:** ใช้แบบบันทึกเหตุการณ์เดิมเป็นแหล่งข้อมูลหลัก แล้วสรุปสถานการณ์ตามรอบเวลา

**ส่วนข้อมูลหลัก:** รอบรายงาน, ช่วงเวลาข้อมูล, ประเภทและจำนวนเหตุการณ์, พื้นที่ได้รับผลกระทบ, ผู้ได้รับผลกระทบ/ผู้ป่วย/ผู้เสียชีวิต (ตามประเภท EOC), จุดเสี่ยง, แนวโน้มเทียบรอบก่อน, แผนที่หรือพิกัด, แหล่งข้อมูล และระดับความเชื่อมั่นของข้อมูล

**สถานะรับข้อมูลรายทีม:** ทีมต้นทาง, เวลาได้รับข้อมูลล่าสุด, สถานะ `received/pending/late`, หมายเหตุการตรวจสอบ และผู้ทวนสอบ

### 10.4 PLANNING - แผนปฏิบัติการและแผนเผชิญเหตุ
**วัตถุประสงค์:** จัดทำและติดตามแผนปฏิบัติการตามรอบเหตุการณ์

**ส่วนข้อมูลหลัก:** ชื่อแผน, รอบปฏิบัติการ, วัตถุประสงค์, สมมติฐาน/สถานการณ์คาดการณ์, พื้นที่เป้าหมาย, กลยุทธ์, กิจกรรม, ทีมรับผิดชอบ, วันเวลาเริ่ม-สิ้นสุด, ตัวชี้วัดความสำเร็จ, ทรัพยากรที่ต้องใช้ และความเสี่ยงของแผน

**ตารางกิจกรรม:** ลำดับ, กิจกรรม, เจ้าภาพ, ผู้สนับสนุน, กำหนดส่ง, สถานะ, ผลจริง และประเด็นต้องตัดสินใจ

### 10.5 SCIENTIFIC - ข้อเสนอแนะวิชาการ
**วัตถุประสงค์:** รวบรวม วิเคราะห์ และรับรองข้อเสนอทางวิชาการสำหรับการสั่งการ

**ส่วนข้อมูลหลัก:** หัวข้อวิชาการ, คำถามที่ต้องตอบ, บทสรุปหลักฐาน, แหล่งอ้างอิง/URL/เอกสาร, ระดับคุณภาพหลักฐาน, ข้อเสนอแนะ, กลุ่มเป้าหมาย, ผลกระทบหากไม่ดำเนินการ, ผู้จัดทำ, ผู้ทบทวน และสถานะรับรอง

**ความเห็นทบทวน:** ผู้ทบทวน, วันที่, ความเห็น, ผลการพิจารณา `approved/revise/rejected`

### 10.6 JIT - รายงานสอบสวนและควบคุมโรค/เหตุการณ์
**วัตถุประสงค์:** บันทึกการออกปฏิบัติการสอบสวน ควบคุมเหตุ และมาตรการในพื้นที่

**ส่วนข้อมูลหลัก:** เลขที่ภารกิจ, วันเวลาแจ้งเหตุและออกปฏิบัติการ, จุดเกิดเหตุ/พิกัด, ประเภทเหตุ, นิยามผู้เกี่ยวข้อง, จำนวนผู้สัมผัสหรือผู้ได้รับผลกระทบ, ผลการสอบสวนเบื้องต้น, มาตรการที่ดำเนินการ, หน่วยร่วมปฏิบัติ, ข้อค้นพบ, ข้อเสนอแนะ และสถานะปิดภารกิจ

**ตารางมาตรการ:** มาตรการ, กลุ่มเป้าหมาย, จำนวนเป้าหมาย, จำนวนดำเนินการ, ผลลัพธ์, ผู้รับผิดชอบ, วันนัดติดตาม

### 10.7 ACTIVE_SURV - รายงานเฝ้าระวังเชิงรุก
**วัตถุประสงค์:** ติดตามการคัดกรองและค้นหาผู้มีความเสี่ยงในพื้นที่เป้าหมาย

**ส่วนข้อมูลหลัก:** วันและรอบการลงพื้นที่, พื้นที่/พิกัด, กลุ่มเป้าหมาย, เกณฑ์คัดกรอง, จำนวนเป้าหมาย, จำนวนคัดกรอง, จำนวนเข้าเกณฑ์, จำนวนส่งต่อ, จำนวนเก็บตัวอย่าง (ถ้ามี), ผลการค้นหา, ช่องว่างการเข้าถึง และแผนติดตาม

**ตารางพื้นที่:** หมู่บ้าน/สถานประกอบการ, เป้าหมาย, คัดกรองแล้ว, พบความเสี่ยง, ส่งต่อ, ผู้รับผิดชอบ, หมายเหตุ

### 10.8 CASE_MGMT - รายงานจัดการผู้ป่วยและระบบส่งต่อ
**วัตถุประสงค์:** สรุปศักยภาพบริการ จำนวนผู้ป่วย และการส่งต่อ โดยไม่เก็บข้อมูลระบุตัวบุคคลเกินจำเป็น

**ส่วนข้อมูลหลัก:** รอบรายงาน, หน่วยบริการ, จำนวนผู้ป่วยแยกความรุนแรง, เตียงใช้ได้/ครองเตียง, ICU และอุปกรณ์สำคัญ, จำนวนรับใหม่/จำหน่าย/ส่งต่อ/เสียชีวิต, ปัญหาบริการ, ความต้องการสนับสนุน และผู้ประสานงาน

**รายการส่งต่อ:** รหัสอ้างอิงที่ไม่เปิดเผยตัวตน, หน่วยต้นทาง, หน่วยปลายทาง, ระดับความเร่งด่วน, เวลาแจ้ง-รับ, สถานะ และอุปสรรค

### 10.9 LAB - รายงานศักยภาพและผลตรวจห้องปฏิบัติการ
**วัตถุประสงค์:** ติดตามตัวอย่าง ผลตรวจ ระยะเวลารอ และน้ำยาหรือวัสดุวิกฤต

**ส่วนข้อมูลหลัก:** รอบรายงาน, ห้องปฏิบัติการ, ชนิดการตรวจ, ตัวอย่างรับเข้า, ตรวจแล้ว, รอผล, ผลบวก/ลบ/รอทวน, ระยะเวลาตอบผลเฉลี่ย, ปัญหาคุณภาพ, ศักยภาพคงเหลือ และความต้องการสนับสนุน

**รายการทดสอบ:** ประเภทตัวอย่าง, วิธีตรวจ, จำนวน, ผล, วันที่รับ, วันที่รายงานผล, สถานะยืนยันผล และหมายเหตุ

### 10.10 RISKCOM - รายงานสื่อสารความเสี่ยงและข่าวลือ
**วัตถุประสงค์:** วางแผน เผยแพร่ และประเมินการสื่อสารสาธารณะ

**ส่วนข้อมูลหลัก:** หัวข้อสื่อสาร, ข้อความหลัก, กลุ่มเป้าหมาย, ช่องทางเผยแพร่, วันที่เวลาเผยแพร่, ผู้รับผิดชอบ, ลิงก์/ไฟล์สื่อ, จำนวนเข้าถึง/การมีส่วนร่วม, คำถามหรือข้อกังวล, ผลตอบรับ และข้อเสนอปรับปรุง

**ทะเบียนข่าวลือ:** เนื้อหาข่าวลือ, แหล่งพบ, พื้นที่/กลุ่มเป้าหมาย, ระดับผลกระทบ, ข้อเท็จจริงที่ตรวจสอบแล้ว, การตอบโต้, สถานะ และวันที่ปิดประเด็น

### 10.11 VACCINE - รายงานบริหารจัดการและบริการวัคซีน
**วัตถุประสงค์:** ติดตามแผนบริการ ความครอบคลุม สต็อก และเหตุการณ์ไม่พึงประสงค์หลังรับวัคซีน

**ส่วนข้อมูลหลัก:** รอบบริการ, หน่วยบริการ/พื้นที่, ชนิดวัคซีน, กลุ่มเป้าหมาย, เป้าหมาย, จำนวนได้รับบริการ, ความครอบคลุม, คงคลังต้นงวด-รับเข้า-ใช้-สูญเสีย-คงเหลือ, อุณหภูมิลูกโซ่ความเย็น, AEFI และความต้องการสนับสนุน

**รายการ AEFI:** รหัสอ้างอิง, วันที่รับวัคซีน, วันที่เกิดอาการ, ระดับความรุนแรง, การรักษา/ส่งต่อ, ผลการติดตาม และสถานะรายงาน

### 10.12 MCATT - รายงานช่วยเหลือเยียวยาจิตใจ
**วัตถุประสงค์:** บันทึกบริการด้านสุขภาพจิตแก่ผู้ได้รับผลกระทบโดยใช้ข้อมูลสรุปและรหัสอ้างอิงที่คุ้มครองข้อมูลส่วนบุคคล

**ส่วนข้อมูลหลัก:** วันและพื้นที่ปฏิบัติงาน, เหตุการณ์, กลุ่มเป้าหมาย, จำนวนคัดกรอง, ระดับความเสี่ยง, จำนวนให้การช่วยเหลือเบื้องต้น, จำนวนส่งต่อ, หน่วยรับส่งต่อ, ประเด็นจิตสังคมหลัก, ทีมปฏิบัติการ และแผนติดตาม

**รายการติดตามรายกรณี:** รหัสอ้างอิง, ระดับความเสี่ยง, วิธีช่วยเหลือ, วันนัด, สถานะติดตาม และผลลัพธ์ โดยห้ามบันทึกชื่อหรือเลขบัตรในแบบฟอร์มทีม

### 10.13 POE - รายงานด่านควบคุมโรคระหว่างประเทศ
**วัตถุประสงค์:** ติดตามผู้เดินทาง ยานพาหนะ การคัดกรอง และเหตุการณ์ที่จุดผ่านแดน

**ส่วนข้อมูลหลัก:** ด่าน/ช่องทางเข้าออก, วันเวลาและกะ, จำนวนผู้เดินทางเข้า/ออก, ยานพาหนะ, จำนวนคัดกรอง, จำนวนเข้าเกณฑ์, จำนวนแยกสังเกตอาการ, จำนวนส่งต่อ, มาตรการที่ใช้, เหตุการณ์ผิดปกติ และหน่วยประสาน

**รายการเหตุผิดปกติ:** รหัสเหตุการณ์, ประเภท, ประเทศ/เส้นทางต้นทาง, มาตรการ, ปลายทางส่งต่อ, สถานะ และหมายเหตุ

### 10.14 QUARANTINE - รายงานสถานที่กักกันและการติดตาม
**วัตถุประสงค์:** ติดตามศักยภาพการกักกัน ผู้เข้าพัก และมาตรการควบคุม โดยใช้ข้อมูลรวม

**ส่วนข้อมูลหลัก:** สถานที่กักกัน, วันรายงาน, ความจุ, ผู้เข้ากักกันปัจจุบัน, รับใหม่, ออกจากการกักกัน, ผู้มีอาการ, ส่งต่อ, บุคลากรปฏิบัติงาน, PPE/เวชภัณฑ์คงเหลือ, ความพร้อมด้านอาหาร-น้ำ-ขยะ และปัญหาเร่งด่วน

**ทะเบียนติดตาม:** รหัสอ้างอิง, วันที่เริ่ม/สิ้นสุด, สถานะการติดตามอาการ, เหตุส่งต่อ, ผลการประเมิน และสถานะปล่อยตัว

### 10.15 MERT - รายงานปฏิบัติการฉุกเฉินทางการแพทย์
**วัตถุประสงค์:** สรุปการออกทีม การดูแล ณ จุดเกิดเหตุ และการส่งต่อผู้ป่วยฉุกเฉิน

**ส่วนข้อมูลหลัก:** เลขที่ภารกิจ, วันเวลาแจ้ง-ถึงที่เกิดเหตุ-สิ้นสุด, จุดเกิดเหตุ/พิกัด, ประเภทเหตุ, จำนวนผู้ป่วยแยกระดับความรุนแรง, การรักษาเบื้องต้น, การส่งต่อ, หน่วยปลายทาง, ยานพาหนะ/ทีมที่ใช้, ระยะเวลาตอบสนอง, ความปลอดภัยทีม และอุปสรรค

**ตารางทรัพยากรภารกิจ:** เจ้าหน้าที่, ยานพาหนะ, อุปกรณ์สำคัญ, เวลาใช้งาน, สถานะคืนพร้อมใช้ และหมายเหตุ

### 10.16 SeRHT - รายงานอนามัยสิ่งแวดล้อมฉุกเฉิน
**วัตถุประสงค์:** ประเมินและจัดการความเสี่ยงน้ำ อาหาร สุขาภิบาล ขยะ และสิ่งแวดล้อมในพื้นที่เหตุการณ์

**ส่วนข้อมูลหลัก:** พื้นที่/สถานประกอบการ, วันสำรวจ, ประเด็นประเมิน, ผลตรวจน้ำหรืออาหาร (ถ้ามี), สุขาภิบาล, ขยะ/สิ่งปฏิกูล, สารเคมี/สิ่งคุกคาม, ระดับความเสี่ยง, มาตรการแก้ไข, หน่วยรับผิดชอบ, วันติดตาม และรูปถ่าย/พิกัด

**รายการความเสี่ยง:** ประเภท, รายละเอียด, ระดับ `low/medium/high/critical`, ผู้ได้รับผลกระทบโดยประมาณ, มาตรการทันที, สถานะติดตาม

### 10.17 LOGISTICS - รายงานคลังและการส่งกำลังบำรุง
**วัตถุประสงค์:** บริหารคำขอ เบิกจ่าย คงคลัง และการจัดส่งทรัพยากร EOC

**ใบขอรับการสนับสนุน:** หน่วยขอ, พื้นที่ใช้, ความเร่งด่วน, รายการ/หน่วยนับ/จำนวน, เหตุผล, วันที่ต้องการ, ผู้ขอ ไม่ต้องมีผู้อนุมัติ

**ใบจ่ายและส่งมอบ:** เลขอ้างอิงคำขอ, รายการ, จำนวนอนุมัติ/จ่ายจริง, คลังต้นทาง, ผู้จ่าย, ผู้รับ, วันเวลาและวิธีส่งมอบ, หลักฐานรับมอบ, สถานะจัดส่ง

**รายงานคงคลัง:** รหัสรายการ, ชื่อรายการ, หน่วย, จุดสั่งซื้อ, คงเหลือ, คงคลังจอง, อัตราใช้เฉลี่ย, จำนวนวันคงเหลือ, วันหมดอายุ/ล็อต และสถานะเสี่ยงขาด

### 10.18 LEGAL - รายงานประเด็นกฎหมายและเอกสารคำสั่ง
**วัตถุประสงค์:** รับเรื่อง วิเคราะห์ข้อกฎหมาย และติดตามเอกสาร/คำสั่งที่เกี่ยวกับเหตุการณ์

**ส่วนข้อมูลหลัก:** เลขที่เรื่อง, ประเภทเรื่อง, ข้อเท็จจริง, คำถามทางกฎหมาย, กฎหมาย/ระเบียบอ้างอิง, ความเห็นทางกฎหมาย, ระดับความเร่งด่วน, หน่วยงานเจ้าของเรื่อง, ผู้รับผิดชอบ, กำหนดตอบ, สถานะ และเอกสารแนบ

**ทะเบียนเอกสาร:** ชนิดเอกสาร, เลขที่, วันที่, ผู้ลงนาม, สถานะตรวจทาน, ลิงก์ไฟล์ และวันหมดอายุ/ทบทวน

### 10.19 FINANCE - รายงานงบประมาณและค่าใช้จ่ายเหตุการณ์
**วัตถุประสงค์:** ควบคุมงบประมาณ คำขอใช้จ่าย ภาระผูกพัน และการเบิกจ่ายตาม Session

**ส่วนข้อมูลหลัก:** แหล่งงบประมาณ, แผนงาน/โครงการ, หมวดค่าใช้จ่าย, วงเงินตั้งต้น, วงเงินขอใช้, อนุมัติแล้ว, ผูกพันแล้ว, เบิกจ่ายแล้ว, คงเหลือ, หน่วยขอ, เหตุผล, เลขที่เอกสารอ้างอิง, ผู้อนุมัติ และสถานะ

**รายการค่าใช้จ่าย:** วันที่, รายการ, ผู้ขาย/หน่วยงาน, จำนวนเงิน, ภาษี, วิธีจัดหา, เอกสารประกอบ, สถานะตรวจสอบ และหมายเหตุ

### 10.20 STAFFING - รายงานจัดสรรกำลังคนและเวรปฏิบัติ
**วัตถุประสงค์:** จัดเวร ติดตามกำลังคนและช่องว่างบุคลากรในพื้นที่/ทีมภารกิจ

**ส่วนข้อมูลหลัก:** รอบเวร, วันเวลาเริ่ม-สิ้นสุด, ทีม/จุดปฏิบัติงาน, จำนวนที่ต้องการ, จำนวนที่จัดได้, รายชื่อหรือรหัสเจ้าหน้าที่, บทบาท, ทักษะ/ใบอนุญาตจำเป็น, สถานะยืนยัน, การเดินทาง/ที่พัก, ผู้ประสาน และช่องว่างกำลังคน

**คำขอกำลังเสริม:** หน่วยขอ, เหตุผล, คุณสมบัติ, จำนวน, พื้นที่, ระยะเวลาปฏิบัติ, ความเร่งด่วน, สถานะจัดหา และผลการจัดสรร

### 10.21 ADMIN - รายงานธุรการและเอกสารสนับสนุน
**วัตถุประสงค์:** จัดการหนังสือรับ-ส่ง การประชุม และงานธุรการของ EOC

**ส่วนข้อมูลหลัก:** ประเภทงาน, เลขรับ/ส่ง, เรื่อง, หน่วยงานต้นทาง/ปลายทาง, วันที่รับ/ส่ง, ระดับความลับ/เร่งด่วน, ผู้รับผิดชอบ, กำหนดตอบ, สถานะดำเนินการ, ลิงก์เอกสาร และหมายเหตุ

**บันทึกการประชุม:** ชื่อการประชุม, วันเวลา, สถานที่/ช่องทางออนไลน์, ผู้เข้าร่วม, วาระ, มติ, งานมอบหมาย, ผู้รับผิดชอบ, กำหนดส่ง และไฟล์รายงานการประชุม

### 10.22 ITSUPPORT - รายงานเหตุขัดข้องและการสนับสนุนดิจิทัล
**วัตถุประสงค์:** ติดตามความพร้อมระบบ เครือข่าย อุปกรณ์ และเหตุขัดข้องที่กระทบ EOC

**ส่วนข้อมูลหลัก:** เลขที่ ticket, วันเวลาแจ้ง, ระบบ/อุปกรณ์, พื้นที่, ผู้แจ้ง, รายละเอียดปัญหา, ผลกระทบ, ระดับความรุนแรง, ผู้รับผิดชอบ, การแก้ไขชั่วคราว, สาเหตุราก, การแก้ไขถาวร, วันเวลาเริ่ม-กู้คืน และสถานะ

**รายงานความพร้อม:** รายการระบบ, สถานะ `available/degraded/unavailable`, เวลา uptime, การสำรองข้อมูลล่าสุด, ความเสี่ยง, แผนแก้ไข และผู้ตรวจสอบ

### 10.23 LIAISON - รายงานประสานงานและข้อตกลงกับหน่วยงาน
**วัตถุประสงค์:** บันทึกการประสานขอรับ/ส่งต่อข้อมูล ทรัพยากร และข้อผูกพันกับหน่วยงานภายนอก

**ส่วนข้อมูลหลัก:** หน่วยงานคู่ประสาน, ชื่อผู้ติดต่อและช่องทาง, เรื่องประสาน, ความต้องการ/ข้อเสนอ, วันเวลาเริ่มประสาน, ผู้รับผิดชอบ, ผลตอบรับ, ข้อตกลง, งานที่ต้องติดตาม, กำหนดส่ง และสถานะ

**ทะเบียนข้อตกลง:** หัวข้อ, หน่วยงาน, ข้อตกลง, ผู้รับผิดชอบทั้งสองฝ่าย, ระยะเวลา, เอกสารอ้างอิง, สถานะ และผลการติดตาม

### 10.24 EOC_MGMT - รายงานการบริหารศูนย์ปฏิบัติการภาวะฉุกเฉิน
**วัตถุประสงค์:** ติดตามความพร้อมของศูนย์ EOC การเปิดใช้งาน ระบบสนับสนุน และประเด็นภาพรวมที่ต้องแก้ไข

**ส่วนข้อมูลหลัก:** รอบรายงาน, สถานะการเปิด EOC, สถานที่/ช่องทางประชุม, จำนวนทีมที่เปิดใช้งาน, สถานะการส่งรายงานแต่ละทีม, บุคลากรประจำศูนย์, ความพร้อมห้องปฏิบัติการ/สื่อสาร/IT, ปัญหาการดำเนินงาน, ความต้องการสนับสนุน และข้อเสนอผู้บัญชาการ

**ตารางติดตามทีม:** ทีม, ผู้ประสาน, สถานะรายงานล่าสุด, เวลาอัปเดตล่าสุด, ประเด็นค้าง, ระดับความเร่งด่วน, ผู้รับผิดชอบแก้ไข และสถานะปิดประเด็น

## 11) จุดที่ต้องยืนยันก่อนสร้างจริง
- กำหนดรอบรายงานมาตรฐานของแต่ละประเภท EOC เช่น ทุก 4/6/12/24 ชั่วโมง หรือรายวัน
- ระบุฟิลด์บังคับและบทบาทผู้อนุมัติของแต่ละทีม โดยเฉพาะคำสั่ง, งบประมาณ, การส่งต่อ และข้อมูลห้องปฏิบัติการ
- ยืนยันระดับข้อมูลส่วนบุคคลที่อนุญาตใน CASE_MGMT, MCATT, QUARANTINE, MERT และ POE; ค่าเริ่มต้นควรใช้ข้อมูลสรุปหรือรหัสอ้างอิงแทนข้อมูลระบุตัวบุคคล
- ยืนยันว่า SAT จะเชื่อมแบบอ่าน/สรุปจากตารางบันทึกเหตุการณ์เดิม หรือจำเป็นต้องมีแบบ Situation Report แยกต่างหาก
- ยืนยัน workflow สถานะกลาง: ผู้บันทึกสร้างฉบับร่าง -> หัวหน้าทีมส่ง/รับรอง -> ผู้เกี่ยวข้องอนุมัติ (เฉพาะทีมที่ต้องมีการอนุมัติ)
