# Prompt: ปรับระบบให้ข้อมูลปฏิบัติการมาจาก Database และแสดงแหล่งข้อมูลอย่างชัดเจน

## บทบาท

คุณเป็น Senior Full-stack Engineer ที่รับผิดชอบระบบ STN-EOC ซึ่งเป็นระบบรายงานและติดตามสถานการณ์ฉุกเฉิน ข้อมูลที่แสดงต่อเจ้าหน้าที่ ผู้บัญชาการ และประชาชนต้องตรวจสอบย้อนกลับได้ ห้ามสร้างข้อมูลปฏิบัติการขึ้นเองเมื่อฐานข้อมูลไม่มีข้อมูล

## เป้าหมาย

ปรับระบบให้ข้อมูลรายงาน สถิติ ประกาศ ระดับความเสี่ยง วันที่ของ Session และข้อมูลที่มีผลต่อการตัดสินใจ มาจาก Database เป็น Source of Truth เดียว

ข้อมูลจาก API ภายนอก เช่น Open-Meteo, GISTDA, OSRM และ OpenStreetMap ใช้ได้เฉพาะกรณีที่เป็นข้อมูลจากระบบภายนอกโดยเจตนา แต่ต้องระบุแหล่งข้อมูล เวลาอัปเดต และสถานะของข้อมูลอย่างชัดเจน ห้ามนำข้อมูลภายนอกหรือข้อมูลคำนวณมาแสดงเสมือนเป็นข้อมูลจาก Database

เมื่อไม่พบข้อมูลจริง ให้แสดง Empty State ว่า “ยังไม่มีข้อมูล” ห้ามใช้ mock, seed, sample, fallback, ค่าคาดเดา หรือข้อมูล hardcode ทดแทน

## กฎก่อนแก้ไข

1. อ่าน `prompter/structure.md` ทั้งไฟล์ก่อนเริ่มงาน และยึดโครงสร้าง Authentication, Permission, Sidebar, Route, Layout และ API เดิม
2. ตรวจ `git status` และรักษาการแก้ไขเดิมของผู้ใช้ ห้ามลบหรือเขียนทับงานที่ไม่เกี่ยวข้อง
3. ตรวจ schema และ migration ที่มีอยู่ก่อนสร้างตารางหรือคอลัมน์ใหม่
4. ใช้ `lib/db` หรือ database abstraction กลางของระบบ ห้ามสร้าง connection pool ซ้ำในแต่ละ route หากสามารถใช้ของเดิมได้
5. ทุก API ต้องตรวจ Authentication, Session membership และ Permission ฝั่ง Server
6. Session ที่ปิดแล้วต้องยังอ่านและเพิ่มรายงานได้ตาม permission `canReport` ที่กำหนดไว้ ห้ามผูกสิทธิ์รายงานกับสถานะ active เพียงอย่างเดียว
7. ห้ามเปลี่ยนข้อมูลจาก Database เป็นข้อมูลจำลองเพื่อให้หน้าจอดูมีข้อมูล

## หลักการแยกประเภทข้อมูล

ข้อมูลต่อไปนี้ไม่จำเป็นต้องอยู่ใน Database:

- สี ธีม icon และ CSS class
- label สำหรับ enum
- route path
- pagination default
- validation rule
- permission key
- ข้อความคำแนะนำทั่วไปที่ระบุชัดว่าไม่ใช่สถานการณ์ปัจจุบัน

ข้อมูลต่อไปนี้ต้องมาจาก Database หรือ external provider ที่ระบุแหล่งที่มา:

- รายงานเหตุการณ์
- จำนวนผู้ได้รับผลกระทบ
- จำนวนผู้ป่วย ผู้เสียชีวิต ผู้รักษาหาย และผู้รักษาในโรงพยาบาล
- ระดับน้ำ ระดับความเสี่ยง และสถานะพื้นที่
- ประกาศ ข่าว และคำเตือนปัจจุบัน
- สมาชิก กลุ่มภารกิจ ภารกิจ บันทึกการประชุม และบันทึกข้อสั่งการ
- ช่วงวันที่ของ Session
- ประชากรที่ใช้เป็นตัวหาร
- Polygon และ master geography
- สถานะถนนและเส้นทาง
- metadata ของไฟล์รายงาน

## Database และ Migration

1. Migration ต้องใช้ `CREATE TABLE IF NOT EXISTS` หรือแนวทาง migration ที่โครงการใช้อยู่
2. ห้ามแก้ migration เก่าที่ถูกใช้งานแล้ว ให้สร้าง migration ใหม่สำหรับการเปลี่ยน schema
3. เพิ่ม foreign key และ index ตาม query จริง
4. ห้ามสร้าง seed ที่มีตัวเลขสถานการณ์จำลองใน production migration
5. Master data ต้องบันทึก source และปี/เวลาอ้างอิง
6. ตรวจตาราง `missions`, `meeting_notes` และ `decision_logs` จาก `migrations/create_flood_eoc_management.sql`
7. หากตารางยังไม่มีใน Database runtime ให้สร้าง migration/installation path ที่ชัดเจนและเชื่อม API จริง
8. ห้ามแสดงข้อมูล mission, meeting หรือ decision จาก mock เมื่อไม่มีตารางหรือไม่มีข้อมูล

## รูปแบบ API มาตรฐาน

API ที่คืนข้อมูลปฏิบัติการควรมี metadata ในรูปแบบใกล้เคียง:

```json
{
  "success": true,
  "data": [],
  "meta": {
    "source_type": "database",
    "source_name": "table_name",
    "session_id": 1,
    "report_date": "YYYY-MM-DD",
    "observed_at": null,
    "generated_at": "ISO-8601",
    "is_calculated": false,
    "data_quality_warning": null
  }
}
```

หากไม่มีข้อมูล:


## Permission และ Closed Session

- `canView`: ดูข้อมูลได้
- `canOperate`: แก้ไขโครงสร้าง Session ทีม และการมอบหมายงาน
- `canReport`: เพิ่มและแก้ไขรายงานตามขอบเขตทีม แม้ Session ปิดแล้ว หากนโยบายอนุญาต
- `canSubmit`: ส่งรายงาน
- `canReview`: ตรวจ รับรอง ตีกลับ หรืออนุมัติรายงาน

ทุก permission ต้องคำนวณจาก Server และส่งให้ UI ผ่าน API ห้ามให้ UI เดาจาก role เอง

## การทดสอบ

เพิ่มหรือปรับ automated tests ให้ครอบคลุมอย่างน้อย:

1. Flood Session 3 ไม่มีข้อมูลแล้วต้องคืน Empty State ไม่ใช่ seed
2. ไม่มีประชากรแล้ว morbidity rate ต้องเป็น `null`
3. ไม่มีข้อมูลผู้เสียชีวิตแล้วต้องไม่รายงานเป็นศูนย์โดยไม่มีคำอธิบาย
4. ไม่มีประกาศแล้วหน้าแรกต้องไม่แสดงประกาศ static
5. Disease Daily Risk ต้องคืนเฉพาะวันที่จาก Database
6. Closed Session ยังรายงานได้เมื่อ `canReport=true`
7. ผู้ไม่มี membership หรือ permission ต้องถูกปฏิเสธฝั่ง Server
8. Polygon district/tambon ต้องมาจาก spatial Database
9. Provider ภายนอกล้มเหลวแล้วต้องไม่แสดง mock
10. Production source ต้องไม่ import ไฟล์ mock หรือ test fixture
11. API list ไฟล์ต้องใช้ metadata จาก Database
12. `missions`, `meeting_notes` และ `decision_logs` ต้องอ่านเขียน Database จริงและแสดง Empty State เมื่อไม่มีข้อมูล

## การตรวจรับ

งานถือว่าเสร็จเมื่อ:

- ไม่พบ `data_mode: "mock"` หรือ `data_source: "google_sheet_seed"` ใน production runtime
- ไม่พบข้อมูลเหตุการณ์ สถิติ หรือประกาศตัวอย่างใน production UI
- ไม่มี fallback ที่สร้างตัวเลขสถานการณ์ขึ้นเอง
- ข้อมูลที่คำนวณมี source และสถานะ `calculated` ชัดเจน
- ข้อมูลภายนอกมี provider และเวลาอัปเดต
- ข้อมูลว่างแสดง Empty State
- แบบฟอร์มพื้นที่ใช้ master data API กลาง
- Session ที่ปิดแล้วยังรายงานได้ตาม permission
- ตาราง mission, meeting และ decision เชื่อมต่อ API และ Database จริง
- Test, lint และ build ผ่าน
- จัดทำสรุปไฟล์ที่แก้ migration ที่เพิ่ม ตารางที่ใช้ และผลการทดสอบ

## ลำดับการดำเนินงาน

1. สำรวจ schema, migration, route และผู้เรียกใช้จริง
2. ทำ inventory แหล่งข้อมูลเป็น `database`, `external_provider`, `calculated`, `configuration`, `filesystem` และ `mock`
3. แก้ข้อมูล mock/seed ที่กระทบการตัดสินใจก่อน
4. สร้าง migration และ repository/service กลาง
5. ปรับ API
6. ปรับ UI และ Empty State
7. ลบ mock ที่ไม่ใช้งานหรือย้ายไป test fixture
8. เพิ่ม tests
9. รัน lint, test และ build
10. รายงานผลพร้อมข้อจำกัดที่ยังเหลือ ห้ามอ้างว่างานสมบูรณ์หากยังมี production fallback เป็นข้อมูลจำลอง
