# งาน: ปรับระบบรายงานเจ้าหน้าที่ให้จบภายในเมนูกลุ่มภารกิจ

## เป้าหมาย

ปรับระบบศูนย์ปฏิบัติงานเจ้าหน้าที่ EOC ให้เจ้าหน้าที่รายงานงานของตนได้ครบถ้วนจากเมนู **กลุ่มภารกิจ** ที่ได้รับมอบหมาย โดยไม่ต้องออกไปใช้หน้าผู้ดูแลระบบหรือโมดูลกลาง และให้ผู้บัญชาการติดตามสถานะการส่งรายงานของแต่ละทีมได้จากข้อมูลจริง

เส้นทางหลักที่ต้องรองรับคือ:

`/eoc/staff` -> เลือกเหตุการณ์ EOC -> เลือกกลุ่มภารกิจ -> `/eoc/staff/[sessionId]/teams/[sessionTeamId]` -> แท็บรายงาน -> บันทึกร่าง/ส่งรายงาน/ดูประวัติของทีม

## ปัญหาปัจจุบันที่ต้องแก้

1. หน้า `/eoc/staff/[sessionId]/teams/[sessionTeamId]` ใช้ `REPORT_LINKS` แบบ hard-code และพาออกไปหน้ากลาง เช่น `/eoc/flood/records`, `/resources/medical-inventory`, `/admin/announcements`
2. รองรับเพียงบาง `team_code` และทีมอื่นไม่มีแบบฟอร์มรายงาน
3. ข้อมูลรายงานอุทกภัยน้ำท่วมผูกเพียง `session_id` แต่ไม่ผูก `session_team_id` หรือ `team_code`
4. API บันทึกข้อมูลยังไม่ตรวจว่าเจ้าหน้าที่เป็นสมาชิกของกลุ่มภารกิจที่กำลังรายงาน
5. API ยังอนุญาตแก้ไขข้อมูลของเหตุการณ์ที่ปิดแล้ว และไม่จำกัดการแก้ไขตามเจ้าของรายงาน/หัวหน้าทีม
6. หน้าจัดการ EOC อุทกภัยบางส่วนใช้ข้อมูล mock จึงไม่สะท้อนสถานะรายงานจริง

## ขอบเขตงาน

### 1. พื้นที่ทำงานกลุ่มภารกิจ

ปรับหน้า `/eoc/staff/[sessionId]/teams/[sessionTeamId]` ดังนี้

- คงแท็บภาพรวม, รายงานผล, ประวัติ
- แท็บ **รายงานผล** ต้องแสดงแบบฟอร์มของกลุ่มภารกิจปัจจุบันภายในหน้านี้ ไม่ใช้ลิงก์ไปหน้า `/admin/*` หรือ `/resources/*`
- รับบริบทจาก URL ได้แก่ `sessionId` และ `sessionTeamId` เสมอ ห้ามให้ผู้ใช้เลือกทีมอื่นเองในแบบฟอร์ม
- แสดงชื่อเหตุการณ์, ชื่อกลุ่มภารกิจ, วันรายงาน, สถานะรายงาน, ผู้บันทึกล่าสุด และเวลาปรับปรุงล่าสุด
- รองรับสถานะอย่างน้อย `draft`, `submitted`, `needs_revision`, `reviewed`, `approved`
- เมื่อเหตุการณ์ EOC ปิดแล้ว ให้แสดงข้อมูลแบบอ่านอย่างเดียวและปิดปุ่มบันทึก/ส่งรายงาน

### 2. แบบฟอร์มรายงานตามกลุ่มภารกิจ

สร้างระบบกำหนดแบบฟอร์มจาก `team_code` โดยไม่ hard-code เป็นลิงก์ออกนอกหน้า

- `SAT`: สถานการณ์อุทกภัยน้ำท่วม, พื้นที่, ระดับน้ำ, ผู้ได้รับผลกระทบ, ความต้องการช่วยเหลือ
- `LOGISTICS`: ทรัพยากร/เวชภัณฑ์, รับเข้า, เบิกจ่าย, คงเหลือ, รายการขาดแคลน
- `RISKCOM`: ข้อความสื่อสารความเสี่ยง, ช่องทางเผยแพร่, กลุ่มเป้าหมาย, เวลาที่เผยแพร่
- ทีมอื่น: ใช้แบบฟอร์มมาตรฐานที่มีหัวข้อ สรุปสถานการณ์, การดำเนินงาน, ปัญหาอุปสรรค, ความต้องการสนับสนุน, เอกสารหรือรูปภาพแนบ

ออกแบบ schema หรือ config ของฟอร์มให้เพิ่ม `team_code` ใหม่ได้โดยไม่ต้องสร้างหน้าใหม่ทุกทีม

### 3. โครงสร้างข้อมูลและ API

สร้างตารางรายงานกลาง เช่น `eoc_team_reports` และ migration ที่ idempotent โดยต้องมีฟิลด์อย่างน้อย:

- `id`
- `eoc_session_id`
- `session_team_id`
- `team_code`
- `report_date`
- `report_type`
- `status`
- `payload` แบบ JSON สำหรับข้อมูลเฉพาะของแต่ละทีม
- `submitted_by`, `submitted_at`
- `reviewed_by`, `reviewed_at`, `review_comment`
- `approved_by`, `approved_at`
- `created_at`, `updated_at`

หากจำเป็นต้องมีรายการทรัพยากรหรือข้อมูลพื้นที่ที่ค้นหา/สรุปบ่อย ให้สร้าง child table เพิ่มเติม แต่รายงานหลักต้องอ้างอิง `session_team_id` เสมอ

สร้าง API ภายใต้แนวทาง เช่น:

- `GET /api/eoc/sessions/[sessionId]/teams/[sessionTeamId]/reports`
- `POST /api/eoc/sessions/[sessionId]/teams/[sessionTeamId]/reports`
- `PUT /api/eoc/sessions/[sessionId]/teams/[sessionTeamId]/reports/[reportId]`
- `POST /api/eoc/sessions/[sessionId]/teams/[sessionTeamId]/reports/[reportId]/submit`
- `POST /api/eoc/sessions/[sessionId]/teams/[sessionTeamId]/reports/[reportId]/review`

ใช้รูปแบบ response และ helper authentication ของโครงการเดิม

### 4. สิทธิ์และความถูกต้องของข้อมูล

บังคับตรวจสอบที่ API ทุก endpoint ไม่พึ่งเพียง UI

- `admin` และ `commander` ดูรายงานทุกทีมได้
- `staff`, `SAT`, `MCATT`, `SeRHT` ดูและสร้างรายงานได้เฉพาะ `session_team_id` ที่ตนเป็นสมาชิกและยัง active
- สมาชิกทีมแก้ไขได้เฉพาะรายงานร่างของทีมตนเองตามสิทธิ์ที่กำหนด
- หัวหน้าทีมหรือผู้มีสิทธิ์ส่งรายงานต้องเป็นผู้เปลี่ยนสถานะเป็น `submitted`
- ผู้บัญชาการ/ผู้ตรวจสอบที่กำหนดเท่านั้นจึงเปลี่ยนสถานะเป็น `reviewed`, `needs_revision`, `approved` ได้
- ปฏิเสธการสร้าง/แก้ไข/ส่งรายงานเมื่อ `eoc_sessions.status = 'closed'`
- ปฏิเสธ `sessionTeamId` ที่ไม่อยู่ใน `sessionId` ที่ระบุ
- บันทึกผู้กระทำและเวลาทุกครั้ง พร้อม activity log ที่มี `session_id`, `session_team_id`, `team_code`, `report_id`, action และรายละเอียดที่จำเป็น

### 5. แดชบอร์ดและการติดตาม

ปรับหน้าจัดการ EOC อุทกภัยให้ดึงสถานะรายงานจากฐานข้อมูลจริง แทนข้อมูล mock สำหรับส่วนต่อไปนี้:

- ทีมที่ส่งแล้ว/ยังไม่ส่ง/ส่งล่าช้า/รอตรวจสอบ
- ความครบถ้วนของรายงานตามวันและตามทีม
- ผู้ส่งล่าสุดและเวลาส่งล่าสุด
- รายการที่ต้องแก้ไขและหมายเหตุจากผู้ตรวจสอบ

ยังคงข้อมูล mock ได้เฉพาะส่วนที่ไม่มีข้อมูลจริงและต้องมีป้ายระบุชัดเจนว่าเป็นข้อมูลตัวอย่าง ห้ามนำข้อมูล mock มาสรุปสถานะรายงานจริง

### 6. การเชื่อมกับหน้าระบบเดิม

- ห้ามลบหน้าบันทึกน้ำท่วม, คลังเวชภัณฑ์ หรือประกาศเดิม หากยังมีผู้ดูแลระบบใช้งาน
- แต่เมนูรายงานของเจ้าหน้าที่ต้องไม่พาไปหน้าเหล่านั้นโดยตรง
- หากต้องใช้ logic เดิม ให้แยกเป็น service/shared helper แล้วเรียกจาก API รายงานของทีม โดยคงบริบท `eoc_session_id` และ `session_team_id`
- ปรับข้อความ UI ให้เป็นภาษาไทย เช่น “เหตุการณ์ EOC”, “กลุ่มภารกิจ”, “บันทึกร่าง”, “ส่งรายงาน”, “ขอแก้ไข”, “ตรวจสอบแล้ว”

## เกณฑ์ตรวจรับ

1. เจ้าหน้าที่ที่เป็นสมาชิกทีม SAT เข้าเมนูทีมของตนเอง สร้างและส่งรายงานได้โดยไม่ออกจาก URL กลุ่มภารกิจ
2. เจ้าหน้าที่ที่ไม่ได้เป็นสมาชิกทีมเข้าถึง API หรือ URL ของทีมนั้นไม่ได้ และได้รับ `403`
3. เจ้าหน้าที่ทีม LOGISTICS และ RISKCOM รายงานได้จากหน้าเดียวกันโดยไม่ต้องมีสิทธิ์ `admin` หรือ `commander`
4. ทุกข้อมูลรายงานมี `eoc_session_id`, `session_team_id`, `team_code` และผู้กระทำครบถ้วน
5. เหตุการณ์ที่ปิดแล้วไม่สามารถสร้าง แก้ไข หรือส่งรายงานผ่าน API ได้
6. ผู้บัญชาการเห็นสถานะการรายงานของทุกทีมจากข้อมูลจริง และสามารถขอแก้ไข/ตรวจสอบ/อนุมัติได้ตามสิทธิ์
7. มี test อย่างน้อยสำหรับ authorization, การล็อกเหตุการณ์ปิด, การตรวจความสัมพันธ์ระหว่าง session กับ team และการเปลี่ยนสถานะรายงาน
8. รัน lint และ test ที่เกี่ยวข้องผ่านก่อนส่งงาน

## ข้อควรระวัง

- ห้ามเปลี่ยนชื่อ API, route หรือ schema เดิมโดยไม่จำเป็น
- ห้ามอนุญาตให้ client ส่ง `team_code` หรือ `session_team_id` แล้วเชื่อถือทันที ต้องตรวจจากฐานข้อมูลและ URL
- ห้ามแก้ไขหรือ revert การเปลี่ยนแปลงอื่นที่มีอยู่ใน workspace
